# SPRINT 38 — Prescriptive Output: Pattern Library

**Version:** 3.67 → 3.68  
**File:** `god_mode_v3.py`

---

## Goal

SIA builds a pattern library from existing guard signals in the codebase itself.
Every place where a developer correctly validated, authenticated, or guarded
becomes a reusable template. This library powers Sprint 39's fix suggestions.

---

## Root Cause / Motivation

Sprint 34's `sia brief` already finds the closest guard by proximity (same file,
same module). This works but is ad-hoc. Sprint 38 makes it systematic: extract
all canonical guard patterns once, cluster them by structure and intent, and
expose them as a queryable library both internally (for Sprint 39) and externally
(for developers who want to know "what's the right way to do X in this codebase").

---

## Changes

### 1. Pattern Extraction Pass

During `sia scan`, after guard signals are detected, run a pattern extraction pass:

For each node with a confirmed `validation_guard` or `auth_guard` signal:
1. Extract the code window: ±5 lines around the guard trigger
2. Classify the guard type:
   - `regex_validation` — re.match / re.fullmatch / re.search
   - `null_check` — `if not x`, `if x is None`
   - `type_check` — isinstance, type()
   - `range_check` — `if x < 0`, `if len(x) > max`
   - `allowlist_check` — `if x not in allowed_values`
   - `schema_check` — jsonschema.validate, pydantic, marshmallow
   - `auth_decorator` — @whitelist, @login_required, @requires_auth
   - `permission_call` — frappe.has_permission(), request.user.is_authenticated
3. Extract: guard expression, exception raised, error message format, imports used

### 2. Pattern Clustering

Group patterns by:
- Guard type (primary cluster)
- Language (Python / JavaScript / Java)
- Domain hint (field name similarity — e.g. all patterns guarding `*_number` fields cluster together)

Each cluster gets a canonical representative: the pattern with the highest quality
score (clearest expression, most informative error message, most callers).

### 3. Pattern Library Output

New top-level section in the JSON report: `pattern_library`

```json
{
  "pattern_library": [
    {
      "id": "PL-001",
      "guard_type": "regex_validation",
      "language": "python",
      "domain_hint": "insurance_number",
      "source_node": "smartversorgt_crm.api:_polaris_token",
      "source_file": "smartversorgt_crm/api.py",
      "source_lines": [89, 94],
      "code": "if not zip_code or not re.match(r'\\d{5}', zip_code):\n    raise frappe.ValidationError(f'Ungültige PLZ: {zip_code}')",
      "imports_required": ["re", "frappe"],
      "exception_class": "frappe.ValidationError",
      "quality_score": 0.82,
      "usage_count": 1
    }
  ]
}
```

### 4. `sia patterns` Command

New CLI command for developers:

```
sia patterns [--type regex_validation|null_check|...] [--lang python] [--domain <hint>]
```

Lists matching patterns from the last scan's pattern library. Useful for developers
who want to know: "how does this codebase validate phone numbers?"

```
$ sia patterns --domain phone
PL-007  regex_validation  python  (3 usages)
  Source: api/validators.py:142
  Code:   if not re.match(r'\+?\d{7,15}', phone): raise ValueError(...)
```

### 5. Quality Scoring

Pattern quality score (0.0–1.0) based on:
- Has informative error message (not just `raise Exception("invalid")`) → +0.3
- Uses specific exception class (not bare `Exception`) → +0.2
- Includes the invalid value in the error message → +0.2
- Used in multiple places (proven pattern) → +0.3 × min(usage_count/5, 1.0)

Low-quality patterns (score < 0.3) are excluded from suggestions but kept in the library.

## Validation

- Self-analysis: SIA's own codebase must produce ≥5 patterns in the library
  (from its own `isinstance` checks, `raise SystemExit` guards, etc.)
- Frappe bench: library must contain ≥50 patterns across frappe + crm
- smartversorgt_crm: `_polaris_token`'s zip_code guard must appear as PL-00x
  with quality_score ≥ 0.6
- `sia patterns --type regex_validation` must return only regex patterns
- Pattern extraction adds < 10% to total scan time (measured on bench)

---

## What Does NOT Change

- Pattern library is read-only — SIA never modifies source files
- Pattern extraction does not change warning counts or scores
- `sia scan` without patterns flag still works; library is generated by default
  (low overhead) but can be suppressed with `--no-patterns`
